处理多年的数据，我们必须重点关注**数据库查询性能**（避免 N+1 查询）和**大批量数据的处理逻辑**。

在 Ruby on Rails 中，针对“大数据量”的记账软件，推荐采用以下模式：

---

## 1. 高效的数据库统计 (ActiveRecord)
不要把数据全取出来用 Ruby 遍历，要让数据库（PostgreSQL）去算。

```ruby
# 统计过去 5 年，每个月的消费总和
# 这种写法会生成一条高效的 SQL GROUP BY 语句，速度极快
stats = Transaction.where(created_at: 5.years.ago..Time.now)
                   .group("strftime('%Y-%m', created_at)") # SQLite 语法，PG 用 date_trunc
                   .sum(:amount)

# 结果会是一个 Hash: {"2024-01" => 5000.0, "2024-02" => 4500.0 ...}
```

## 2. 导出数据（避免内存爆炸）
当数据量达到几万条时，传统的 `.all.each` 会把所有数据一次性加载到内存，导致程序崩溃。
**必须使用 `find_each`：**

```ruby
require 'csv'

def export_five_years_data
  file_path = "tmp/all_transactions_#{Time.now.to_i}.csv"
  
  CSV.open(file_path, "wb") do |csv|
    csv << ["日期", "类别", "金额", "备注"]

    # find_each 每次只从数据库取 1000 条，处理完再取下一批
    # 这样无论你有 10 万条还是 100 万条数据，内存占用都极低
    Transaction.find_each(batch_size: 1000) do |t|
      csv << [t.created_at, t.category.name, t.amount, t.note]
    end
  end
  
  puts "导出完成：#{file_path}"
end
```

## 3. 前端展示：Hotwire 的局部刷新
数据建议按“年”或“月”进行切片加载。使用 Rails 的 **Turbo Frames**，用户点击“2022 年”时，页面只更新列表区域，无需刷新整个网页。

```erb
<%# app/views/transactions/index.html.erb %>
<h2>我的账单历史</h2>

<%= turbo_frame_tag "yearly_report" do %>
  <%# 这里根据参数展示某一年的数据 %>
  <%= render @transactions %>
  
  <%# 分页链接，使用 Pagy gem %>
  <%== pagy_nav(@pagy) %>
<% end %>
```
